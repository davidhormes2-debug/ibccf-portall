# CI Checks Reference

Full descriptions of every required branch-protection CI check for the IBCCF project. `replit.md` keeps a compact grouped list of check names (grouped by workflow) for quick scanning; this file has the detail — what each check verifies, its test/script file, and how to run or reproduce it locally.

To apply or verify the branch protection rules referenced throughout this file, run `bash scripts/setup-github-protection.sh` / `bash scripts/check-github-protection.sh`.

## Run & Operate
All commands run from the repo root. There is a single `package.json`; `client/` and `server/` are source folders, not separate packages.
- **Dev (client + server on :5000)**: `npm run dev` — tsx runs `server/index.ts`, which mounts the Vite dev middleware.
- **Build / Run prod**: `npm run build` (client → `dist/public/`, server → `dist/index.cjs`), then `npm start`. Runs as CI jobs (required branch protection status check: "Smoke Test / Build") and post-build smoke verification (required branch protection status check: "Smoke Test / Smoke Test"). To apply or verify the branch protection rules, run `bash scripts/setup-github-protection.sh` / `bash scripts/check-github-protection.sh`.
- **Typecheck**: `npm run check`. **Tests**: `npm test` (Vitest — required branch protection status check: "Unit Tests / Vitest"). **DB push**: `npm run db:push` (drizzle-kit from `shared/schema.ts`). Run `bash scripts/db-migrate.sh` first if the push fails with a type-cast error — that script applies idempotent `ALTER TABLE … TYPE … USING` migrations for columns whose type changed after the table was created (e.g. `community_threads.view_count`/`reply_count` text→integer). The migration script is also called automatically by `scripts/post-merge.sh` after every merge. **Schema drift pre-check**: `npm run db:check-drift` (`scripts/check-schema-drift.ts`) uses drizzle-orm introspection (`getTableColumns`/`getSQLType`) to compare every column's declared type in `shared/schema.ts` against `information_schema.columns` in the live database, then exits non-zero if any mismatch is found. This runs automatically in `scripts/post-merge.sh` before `db:push` so type-cast errors are caught and reported with remediation guidance before drizzle-kit ever sees them. When a new mismatch is detected, add an idempotent `ALTER TABLE … TYPE … USING` block to `scripts/db-migrate.sh` and re-run `npm run db:check-drift` to confirm it passes. The pg client's `connectionTimeoutMillis` (default 10000ms) is configurable via the `SCHEMA_DRIFT_CONNECT_TIMEOUT_MS` env var without editing the script — useful for a CI runner on a slower network path to the DB. On a connection failure, the script classifies the underlying pg/node error (via `err.code` and message) into one of: connection refused (fast — nothing listening), DNS failure (hostname unresolved), connection timed out (slow — near the configured ceiling, host likely blackholed/firewalled), or auth failure (bad credentials), so a maintainer debugging a flaky post-merge pipeline can immediately tell which failure mode occurred instead of parsing a raw stack trace. **Migration cast coverage check**: `npm run db:check-migrate-coverage` (`scripts/check-migrate-coverage.ts`) covers two related gaps. (1) A developer changes a column's declared type to `integer`/`boolean` in `shared/schema.ts` but forgets to add the corresponding `USING` cast block to `scripts/db-migrate.sh` — it cross-references every `text`-in-DB / `integer`-or-`boolean`-in-schema column against the cast blocks already present in `db-migrate.sh` and exits non-zero naming the specific `table.column` and type still missing coverage. (2) A developer adds a brand-new column of **any** declared type (text, integer, boolean, varchar, timestamp, jsonb, ...) to an existing table, relying on `db:push` to add it live — but `db:push` can fail or be skipped in some environments, so it also requires a matching `ADD COLUMN IF NOT EXISTS <col> <type>` guard in `db-migrate.sh` (mirroring the existing `is_flagged`/`flag_reason` convention) for every such column, not just integer/boolean ones. Brand-new tables, and brand-new columns already covered by an existing guard, are excluded. Runs automatically in `scripts/post-merge.sh` right after `db:check-drift` and before `db:push`. A single CI guard ("Unit Tests / Schema Drift Guard", `server/__tests__/schemaDriftGuard.test.ts`) statically asserts both scripts exist, use `getTableColumns`/`information_schema.columns`, maps `serial→integer`, excludes brand-new tables/columns, exit non-zero on mismatches, and are wired into `post-merge.sh` before `db:push` — required branch protection status check. The test suite's own TypeScript types are also checked in CI (required branch protection status check: "Unit Tests / Test Suite Type Check"). To apply or verify the branch protection rules, run `bash scripts/setup-github-protection.sh` / `bash scripts/check-github-protection.sh`.
- **Security lint**: `npm run lint:security` — runs `scripts/check-zod-error-leak.sh`, which scans `server/**/*.ts` for raw ZodError array forwarding to clients (e.g. `res.json({ error: error.errors })`). **Never** send a ZodError `.errors` array directly in a response; extract a plain string instead (`"Validation error"` or `errors[0].message`). To suppress a rare false-positive on a specific line, append `// zod-error-leak-ok`. Cross-file helper patterns are out of scope for automated detection (see Gotchas → ZodError leak check scope). Runs as a CI job (required branch protection status check: "Security Lint / Security Lint"). - **i18n key check**: `npm run check:i18n` — runs `scripts/check-i18n.ts` via tsx, which auto-discovers every namespace (all `.json` files in `en/`, excluding `_meta.json`) and diffs the dot-notation key set of every locale (`de`, `es`, `fr`, `pt`, `zh`) against the English reference. Exits non-zero on any missing or extra key so silent `defaultValue` fallbacks are caught before they ship. `npm run i18n:fix` (alias for `tsx scripts/check-i18n.ts --fix`) auto-scaffolds any missing locale namespace files with `[UNTRANSLATED]` stubs for all non-English locales, then runs the full key check — exits 0 so developers can unblock themselves in one step instead of running scaffold + check separately. `npm run i18n:translate` (alias for `tsx scripts/translate-i18n.ts`) fills every `[UNTRANSLATED]` stub across all locale files using OpenAI (requires `AI_INTEGRATIONS_OPENAI_API_KEY` or `OPENAI_API_KEY`); idempotent — only touches stubs, never overwrites existing translations. Supports `--locale de,es`, `--namespace emails.json`, and `--dry-run` flags. After running, verify with `npm run check:untranslated-strict`. Also runs as the `i18n-check` CI job in `.github/workflows/i18n-check.yml` (no path filter — triggers on every push and pull_request, making it eligible as a required branch protection status check: "i18n Key Consistency / i18n Key Consistency"). To apply or verify the branch protection rule, run `bash scripts/setup-github-protection.sh` / `bash scripts/check-github-protection.sh`. `npm run check:untranslated-strict` (alias for `tsx scripts/check-i18n.ts --strict`) runs the full key check AND exits 1 when any `[UNTRANSLATED]` stubs remain.
