---
name: Sandbox dev-server backgrounding for local e2e runs
description: Manual background dev-server processes don't survive between bash tool calls in this project's sandbox; use the managed workflow instead.
---

Starting the dev server manually (`nohup npm run dev > log 2>&1 &`, with or without `disown`/`setsid`) reliably dies or becomes unreachable by the time the next bash tool call runs, even though the process appeared to boot successfully (logged "serving on port 5000", `curl` returned 200) in the same or an immediately following call.

**Why:** Each bash tool invocation appears to run in an isolated shell context in this sandbox; background processes not owned by the platform's own process supervisor get reaped between invocations. This is not specific to this app's code — it reproduced across several different backgrounding techniques (`nohup`, `disown`, `setsid`).

**How to apply:** When you need the dev server running to execute Playwright e2e specs locally (outside of CI), start it via the project's configured Replit workflow (`restart_workflow` on "Start application" or whatever `npm run dev` workflow exists) rather than a manual background shell command. If a live local run still can't be made to work reliably before a time/attempt budget is exhausted, it's acceptable to rely on: (1) a clean typecheck, (2) close structural similarity to an existing passing test, and (3) the CI job that already runs the spec file, rather than repeatedly retrying local process management.

**Playwright specifically:** `npx playwright test` itself is a long-running foreground process, not a persistent server — the "run in background across tool calls" problem still applies. The workaround that reliably works: launch it with `&` and poll it with a `for`/`sleep` loop *inside the same single bash tool call* (e.g. `cmd > log 2>&1 & PID=$!; for i in ...; do sleep 10; kill -0 $PID || break; done; cat log`), sized to the tool's ~120s timeout ceiling. Splitting the launch and the poll across separate tool calls reliably loses the process.

**Admin e2e specs may hang in this sandbox regardless (separate issue):** even with the above technique, every admin-dashboard Playwright spec (verified against both a new spec and the pre-existing, CI-passing `e2e/admin-reactivation-nav-badge-other-tabs.spec.ts` as a control) can time out at the shared `e2e/global-setup.ts` step — the `admin-data-ready` sentinel in `AdminDashboard.tsx` never attaches within its 120s wait. This reproduces identically on a spec already passing in CI, so it's a sandbox/environment limitation, not a spec defect. When this happens, don't keep retrying — rely on typecheck + structural parity with a passing reference spec, and let the CI job be the actual verification.

**Docker-service-backed specs (Mailhog, etc.) hit the same wall, and `/tmp` output doesn't survive a tool-timeout kill either.** Reproduced again with a brand-new spec needing a local Mailhog container: `npx playwright test` (which itself spawns `npm run dev` as its managed webServer) takes longer than the ~120s bash-tool ceiling to install deps/compile/boot/run, so wrapping it in `timeout 110 ... > /tmp/log 2>&1` and letting the *tool's own* hard timeout fire (not the inner `timeout`) doesn't just kill the process — the redirected `/tmp` log file itself is gone by the very next tool call (`No such file or directory`), so there's nothing to inspect after the fact. Don't chase this by writing to `/tmp` and re-reading it later; if a spec can't finish inside one ~120s call, treat it as unrunnable locally in this sandbox and fall back to static verification (typecheck, lint, manual selector-vs-source cross-check, CI sync-check scripts) per the guidance above.

**Update — a *foreground* (not `&`-backgrounded) `timeout N npx playwright test ... > /tmp/log 2>&1` command DOES survive past the tool's own client-side cutoff and stays pollable in later calls**, when the target app server is already running via the platform's own managed workflow (not a manually-started dev server). Confirmed repeatedly: a call returns "Command exited... exit code: -1 / no output" when it hits the ~120s ceiling, but `ps aux` in the very next call still shows the real `npx playwright`/Chromium process tree alive, and `/tmp/log` keeps growing — polling with `sleep N; tail /tmp/log` in follow-up calls eventually captures the full, real pass/fail result. This contradicts the earlier "output doesn't survive" finding above; the difference appears to be that here the *app server* was already up via `restart_workflow`, so the killed call had nothing of its own to tear down beyond the wrapper shell — the child `playwright`/`chrome` processes it spawned were not part of whatever gets reaped. Prior failed attempts specifically used explicit backgrounding (`&`, `nohup`, `disown`, `setsid`) *before* the tool call returned, which did reliably lose the process — plain foreground-with-internal-`timeout` was not tried before and works better. **How to apply:** to observe a Playwright run whose worst-case duration exceeds ~120s, start the app via its managed workflow first, then run the spec as a plain foreground command (`timeout <N> npx playwright test ... > /tmp/log.txt 2>&1`, no trailing `&`) and poll `/tmp/log.txt` with `sleep`+`tail` in subsequent calls instead of trying to background it explicitly. Always `pkill -9 -f chrome-linux`/`chrome_crashpad` afterward regardless of outcome.
